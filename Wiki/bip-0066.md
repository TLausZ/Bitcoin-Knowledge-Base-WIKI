# BIP-0066: Strict DER signatures

## Summary

This document specifies proposed changes to the Bitcoin transaction validity rules to restrict signatures to strict DER encoding.

## Body

**Typ:** Specification  
**Layer:** Consensus (soft fork)  
**Status:** Deployed (aktiviert und im Einsatz)  
**Themen:** bips
**Autor(en):** Pieter Wuille  
**Veröffentlicht:** 2015-01-10  
**Quelle:** https://github.com/bitcoin/bips/blob/master/bip-0066.mediawiki

### Zweck

BIP-0066 required all ECDSA signatures in Bitcoin transactions to use strict DER encoding, activated as a soft fork in July 2015. This closed a specific malleability vector (non-canonical DER encodings could change a signature's byte representation without invalidating it) and also fixed a cross-platform consistency bug.

The bug: Bitcoin used OpenSSL's signature verification, and different OpenSSL versions behaved differently on non-strictly-DER-encoded signatures. Some versions accepted them; others rejected them. This meant nodes running different OpenSSL versions could disagree on whether a transaction was valid — a consensus bug. BIP-66 fixed it by requiring strict DER encoding at the consensus layer, making signature validity independent of OpenSSL's behaviour.

The activation was itself notable: a miner accidentally triggered an unintentional chain split by mining a version-3 block (signalling BIP-66 support) that violated BIP-66's rules. About 6 blocks were orphaned before the network converged. This incident exposed the difference between signalling support and actually running updated validation code during a soft fork deployment.
