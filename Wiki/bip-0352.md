# BIP-0352: Silent Payments

## Summary

This document proposes Silent Payments, a protocol for reusable payment codes enabling privacy-preserving payments without interaction between sender and receiver.

### Zweck

Silent Payments allow a receiver to publish a single static address that each sender uses to derive a unique, one-time payment address. Two different senders to the same static address produce completely unrelated on-chain outputs — there is no common script or pubkey linking them.

The mechanism uses ECDH: the sender combines their private key with the receiver's static public key to derive a shared secret; the receiver scans using their private key to detect payments. No notification transaction is needed; no on-chain data reveals the static address.

Payments to the same recipient from different senders produce outputs that chain analysis cannot link, provided inputs follow the protocol. The receiver's static address never appears on-chain. Two payments to the same recipient look, to any observer, like two completely unrelated transactions.

The scanning requirement is the main practical limitation. Recipients must process every eligible transaction (those with Taproot or SegWit v0 inputs) to detect incoming payments. This is heavier than standard wallet scanning where you derive addresses and check against an index. BIP-392 defines compact block filters to reduce this scanning burden, and BIP-375/376 add PSBT fields for Silent Payment outputs. BIP-353 handles DNS-based human-readable names for static addresses.

As of 2024, no major hardware wallet supports Silent Payments due to the scanning requirement.
