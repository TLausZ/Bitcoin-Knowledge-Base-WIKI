# BIP-0060: Fixed Length "version" Message (Relay-Transactions Field)

## Summary

BIP-37 introduced a new flag to version messages which says whether to relay new transaction messages to that node. This BIP addresses problems with the optional nature of that flag.

## Body

**Typ:** Specification  
**Layer:** Peer Services  
**Status:** Closed (geschlossen (zurückgezogen oder ersetzt))  
**Autor(en):** Amir Taaki  
**Veröffentlicht:** 2013-06-16  
**Quelle:** https://github.com/bitcoin/bips/blob/master/bip-0060.mediawiki

### Zweck

BIP-0060 identified a parsing problem introduced by BIP-37. The `fRelay` flag added to the `version` message was optional — its presence depended on the protocol version — meaning parsers had to check whether the field was present before reading it. This created fragile parsing code where implementations could disagree on whether to expect the field.

BIP-60 proposed making the `version` message fixed-length by always including `fRelay`, defaulting to `true` for old clients that didn't understand it. The proposal was closed without adoption; in practice Bitcoin Core handled the optional field cleanly, and the parsing issue wasn't severe enough to warrant a protocol change.
