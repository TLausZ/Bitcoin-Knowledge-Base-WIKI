# BIP-0443: OP_CHECKCONTRACTVERIFY

## Summary

This BIP proposes OP_CHECKCONTRACTVERIFY (CCV), a tapscript opcode enabling UTXOs that carry dynamic data commitments and constrain how outputs must be structured.

### Zweck

OP_CHECKCONTRACTVERIFY allows a tapscript to: (1) read a "contract data" field embedded in the current input's internal Taproot key, (2) validate or process that data during script execution, and (3) constrain the internal key and taptree of specified outputs to match a required structure (possibly including updated contract data).

This enables "state-carrying UTXOs" — outputs that function like smart contract instances. Each UTXO carries a piece of data (encoded as a Taproot key tweak), and the spending script can both inspect this data and enforce that the output carries updated data according to contract rules.

Vaults are the main use case: the vault UTXO carries the "target address" as its data, the CCV script enforces that a withdrawal must go to that target (no substitution), and the recovery path is always available. This supersedes BIP-345 (which used two dedicated vault opcodes) with a more general primitive.

CCV also enables payment pools, batched channel factories, and other protocols that need to carry state between transactions without a trusted coordinator.
