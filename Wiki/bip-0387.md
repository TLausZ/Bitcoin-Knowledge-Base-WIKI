# BIP-0387: Tapscript Multisig Output Script Descriptors

## Summary

This document specifies `multi_a()` and `sortedmulti_a()` descriptor functions for multisig in Tapscript contexts.

## Body

**Typ:** Informational  
**Layer:** Applications  
**Status:** Deployed (aktiviert und im Einsatz)  
**Themen:** bips, self-custody, privacy
**Autor(en):** Pieter Wuille, Ava Chow  
**Veröffentlicht:** 2024-04-17

### Zweck

BIP-383's `multi()` and `sortedmulti()` use `OP_CHECKMULTISIG`, which is disabled in Tapscript (BIP-342). Tapscript uses `OP_CHECKSIGADD` instead, enabling Schnorr-based threshold signatures with different stack semantics.

BIP-387 defines `multi_a()` and `sortedmulti_a()` for use inside `tr()` leaves. The `_a` suffix indicates these use `OP_CHECKSIGADD`. The script produced by `multi_a(2, KEY1, KEY2, KEY3)` is: `KEY1 OP_CHECKSIG KEY2 OP_CHECKSIGADD KEY3 OP_CHECKSIGADD OP_2 OP_NUMEQUAL`. This is only valid inside a Tapscript context — it cannot be used in `wsh()` or `sh()`.

A typical use: `tr(internal_key, multi_a(2, KEY1, KEY2, KEY3))` — Taproot with a 2-of-3 script-path fallback. The key-path handles the common case (e.g., all 3 signers agree via MuSig2); the `multi_a()` leaf handles recovery if MuSig2 fails.
