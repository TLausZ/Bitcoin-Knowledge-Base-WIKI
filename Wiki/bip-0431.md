# BIP-0431: Topology Restrictions for Pinning

## Summary

This document describes mempool pinning problems and proposes "TRUC" (Topologically Restricted Until Confirmation) transactions — a policy with adjusted topology limits that makes fee-bumping more reliable.

### Zweck

Mempool pinning is an attack on fee-bumping. With RBF (BIP-125), a transaction can be replaced by a higher-fee version. But an attacker (e.g., a Lightning counterparty with a stuck channel) can make replacement expensive or impossible by broadcasting their own descendant transactions that push the combined package size or fee requirements to an extreme. The victim's fee bump has to pay for all the attacker's junk descendants.

TRUC (v3 transactions) addresses this by introducing a new transaction version (nVersion=3) that opts into strict topology limits: a v3 transaction can have at most one unconfirmed parent (no long chains), and a v3 child can have a maximum weight of 1000 vbytes. This limits how much "junk" an attacker can attach. Within these limits, any replacement is straightforward to evaluate: the replacement just needs to pay more fees than the replaced package.

TRUC is designed for Lightning commitment transactions and other pre-signed transaction workflows where pinning is a real adversarial concern. It's deployed in Bitcoin Core and can be combined with Pay-to-Anchor (BIP-433) outputs for CPFP fee bumping.
