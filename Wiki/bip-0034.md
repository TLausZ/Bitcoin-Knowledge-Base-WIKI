# BIP-0034: Block v2, Height in Coinbase

## Summary

Bitcoin blocks and transactions are versioned binary structures. This BIP introduces an upgrade path for versioned transactions and blocks. A unique value is added to newly produced coinbase transactions, and blocks are updated to version 2.

## Body

**Layer:** Consensus (soft fork)  
**Status:** Deployed (aktiviert und im Einsatz)

### Zweck

BIP-0034 required miners to include the block height as the first item in the coinbase script of every block. This may sound minor, but it had a significant security consequence: it made every coinbase transaction unique by construction. Before BIP-34, two coinbase transactions could have the same txid (this actually happened in the blockchain — see BIP-30), allowing a new block's coinbase to overwrite an earlier unspent coinbase in the UTXO set.

By including the block height in the coinbase script, no two coinbase transactions can ever be identical (since block heights are unique). This made BIP-30's explicit duplicate-transaction check redundant for all blocks after the activation height. BIP-34 was activated via miner version signalling: blocks with `version >= 2` had to include the height; once 75% of recent blocks were version 2, all version 2 blocks were required to include it; at 95%, version 1 blocks were rejected.
