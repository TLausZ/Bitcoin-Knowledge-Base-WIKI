# BIP-0073: Use "Accept" header for response type negotiation with Payment Request URLs

## Summary

This BIP describes an enhancement to the payment protocol that allows the client and server to negotiate the content of the response using the HTTP Accept header field.

## Body

**Typ:** Specification  
**Layer:** Applications  
**Status:** Deployed (aktiviert und im Einsatz)  
**Themen:** bips, lightning
**Autor(en):** Stephen Pair  
**Veröffentlicht:** 2013-08-27  
**Quelle:** https://github.com/bitcoin/bips/blob/master/bip-0073.mediawiki

### Zweck

BIP-0073 addresses a UX problem with BIP-72: the `r=` URL in a BIP-72 URI is a dedicated PaymentRequest endpoint that returns a Protocol Buffer, which breaks if opened in a normal browser. Merchants had to maintain two URLs — one for browsers, one for wallets.

BIP-73 solves this with HTTP content negotiation. A single URL can serve either a BIP-70 PaymentRequest (to wallets that send `Accept: application/bitcoin-paymentrequest`) or a human-readable webpage (to browsers). The wallet indicates what it wants via the HTTP Accept header; the server responds appropriately.

This meant merchants only needed one URL on their QR codes, which simplified deployment. The approach has no relevance now that BIP-70 is deprecated.
