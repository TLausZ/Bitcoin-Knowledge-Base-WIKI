# BIP-0157: Client Side Block Filtering

## Summary

This BIP describes a new light client protocol where full nodes generate deterministic compact filters of block content (defined in BIP-158) that are served to clients. A light client downloads an entire block only if the filter matches relevant data.

### Zweck

BIP-0157 defines the P2P protocol for the "neutrino" light client model. It replaces BIP-37 (Bloom filters) as the recommended way for lightweight clients to sync without downloading the full blockchain.

Unlike BIP-37, which sends a filter from client to server, BIP-157/158 reverses this. With Bloom filters, the *client* sends a filter to the *server*, which then sends only matching transactions. This leaks information about the client's addresses to the server. With BIP-157/158, *servers* pre-compute filters for every block (BIP-158 defines the filter format). Clients download these small filters and check locally whether a block might contain their transactions. If the filter matches, they download the full block. If it doesn't, they skip it. Servers learn only which blocks were requested, not which addresses are being checked.

New P2P messages defined in BIP-157: `getcfheaders`, `cfheaders`, `getcfcheckpt`, `cfcheckpt`, and `getckpfilter`/`cfilter`. The filter header chain allows clients to verify filter commitments without downloading all filters.
