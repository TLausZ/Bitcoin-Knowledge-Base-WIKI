# BIP-0053: Disallow 64-byte transactions

## Summary

This BIP describes the rationale for disallowing transactions that are serialized to 64 bytes without the transaction's witness. It addresses weaknesses in the Merkle tree included in Bitcoin block headers.

## Body

**Typ:** Specification  
**Layer:** Consensus (soft fork)  
**Status:** Draft (noch im Entwurfsstadium)  
**Autor(en):** Chris Stewart

### Zweck

BIP-0053 proposes making 64-byte transactions invalid as a soft fork. The motivation is a long-known weakness in Bitcoin's Merkle tree construction: internal Merkle tree nodes are 64 bytes, and a carefully crafted 64-byte transaction could be mistaken for an internal node by a verifier, enabling merkle proof forgery attacks. This is sometimes called the "64-byte transaction" or "CVE-2012-2459" vulnerability class.

The attack matters primarily for SPV clients and proof systems that rely on Bitcoin's Merkle proofs: a malicious node could potentially craft a fake payment proof that a naive SPV client would accept. Bitcoin Core has defences against this, but the underlying ambiguity in the Merkle tree format remains.

Disallowing 64-byte transactions at the consensus layer eliminates the ambiguity entirely. The constraint is very small — 64-byte transactions are effectively impossible to create in normal use anyway — so the cost of the restriction is negligible. BIP-53 was drafted in 2025 alongside discussion of CTV and other script extensions that involve more formal proof systems.
