# BIP-0342: Validation of Taproot Scripts

## Summary

This document specifies the script execution semantics (Tapscript) for BIP-341 Taproot script-path spends.

## Body

**Typ:** Specification  
**Layer:** Consensus (soft fork)  
**Status:** Deployed (aktiviert und im Einsatz)  
**Themen:** bips

**Veröffentlicht:** 2020-01-19

### Zweck

When a Taproot script-path spend is executed, the leaf script is validated under Tapscript rules, which differ from the legacy script and SegWit v0 (P2WSH) rules.

Key changes:
- `OP_CHECKSIG` and `OP_CHECKSIGVERIFY` use Schnorr signatures (BIP-340) instead of ECDSA. This enables future batch validation.
- `OP_CHECKMULTISIG` and `OP_CHECKMULTISIGVERIFY` are disabled (they're incompatible with Schnorr's linearity benefits). Multisig in Tapscript uses `OP_CHECKSIGADD` instead, which is designed for Schnorr threshold signing.
- Unknown opcodes in the range `OP_SUCCESS` (`0x50`, `0x60`, etc.) immediately succeed rather than failing. This provides a soft-fork upgrade path: a future opcode can be defined that makes currently-succeeding scripts fail under new rules, without requiring a new witness version.

The leaf version byte in the Tapscript tree also provides versioning: leaf version 0xc0 specifies Tapscript as defined by BIP-342; other leaf versions are reserved for future script upgrades.
