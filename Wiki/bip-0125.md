# BIP-0125: Opt-in Full Replace-by-Fee Signaling

## Summary

Opt-in full Replace-by-Fee (RBF) signaling allows spenders to signal that they want to be able to replace a transaction in the future, enabling fee bumping on unconfirmed transactions.

### Zweck

BIP-0125 defines opt-in RBF: a transaction signals replaceability by setting at least one input's `nSequence` value to `0xFFFFFFFD` or lower (but not `0xFFFFFFFF`, which means the transaction is final). Nodes that implement BIP-125 will accept a replacement transaction for a signalling transaction if the replacement pays a higher fee rate.

Without RBF, a sender who broadcast a transaction with too low a fee was stuck — the transaction would sit in mempools indefinitely and might never confirm. Opt-in RBF allows the sender to broadcast a new version of the same transaction paying more fees. The replacement spends the same inputs (preventing double-spend of the original), so the recipient's funds aren't at risk, just the timing.

The "opt-in" design was a deliberate compromise: merchants who accepted zero-confirmation payments could safely do so for transactions *without* the RBF signal. Only signalling transactions indicate the sender might replace them. Bitcoin Core has shipped RBF since version 0.12 (2016). A later policy called full-RBF (no signalling required) has been a source of ongoing debate.

## Open Questions

- Does opt-in RBF still provide meaningful zero-confirmation safety for non-signalling transactions, or has full-RBF deployment undermined this guarantee?
- How does CPFP (child-pays-for-parent) compare to RBF as a fee-bumping strategy, and when should each be used?
