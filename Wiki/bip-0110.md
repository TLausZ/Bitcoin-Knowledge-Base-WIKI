# BIP-0110: Reduced Data Temporary Softfork

## Summary

Temporarily limit the size of data fields at the consensus level, in order to correct distorted incentives caused by standardizing support for arbitrary data, and to refocus priorities on improving Bitcoin as money.

## Body

**Typ:** Specification  
**Layer:** Consensus (soft fork)  
**Status:** Draft (noch im Entwurfsstadium)  
**Autor(en):** Dathon Ohm

### Zweck

BIP-0110 is a 2025 Draft that proposes temporarily restricting the amount of arbitrary data that can be embedded in Bitcoin transactions via witness data and other fields. The motivation is the Ordinals/Inscriptions phenomenon: since Taproot's activation in 2021, witness data has been used to store images, text, and other non-financial data in Bitcoin blocks. BIP-110 argues this distorts incentives — miners prioritise data-carrying transactions for their fees, crowding out financial transactions — and proposes a soft fork cap on witness data size per input.

The "temporary" framing is notable: the proposal suggests this as a stopgap while longer-term approaches to data carrier limits are debated. As of early 2026, it remains in Draft with no clear activation path. The proposal is politically contentious: proponents argue data storage on Bitcoin block space is economically harmful; opponents argue Bitcoin's consensus rules should not discriminate by transaction purpose.

## Open Questions

- Is limiting data per witness input enforceable without breaking legitimate Taproot script paths that happen to use large witness stacks?
- Who determines when the "temporary" restriction should expire?
