# BIP-0090: Buried Deployments

## Summary

Prior soft forks (BIP 34, BIP 65, and BIP 66) were activated via miner signaling in block version numbers. This BIP replaces the trigger mechanism by caching the block heights at which those consensus rules became enforced.

### Zweck

Early soft forks (BIP-34, BIP-65, BIP-66) activated via miner version signalling — nodes had to check block version numbers across thousands of blocks to determine whether the rules were active at any given height. By 2016, these activations were ancient history: the rules had been in effect for years, and any node syncing from genesis would spend processing time re-evaluating signalling windows for rules that had been universally adopted for years.

BIP-90 simplified this: instead of re-evaluating the version signalling, Bitcoin Core hardcoded the block heights at which each rule became enforced. A node syncing from scratch simply applies the rules starting at the hardcoded height, skipping the signalling machinery entirely. This speeds up initial block download and removes dead code paths.

The broader significance: BIP-90 established the concept of "burying" a deployment — declaring it permanently part of consensus history without any realistic chance of reorg past its activation. This same principle applies to SegWit, which is now also buried in practice.
