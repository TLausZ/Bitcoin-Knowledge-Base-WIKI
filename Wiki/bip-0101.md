# BIP-0101: Increase maximum block size

## Summary

This BIP proposes replacing the fixed one megabyte maximum block size with a maximum size that grows over time at a predictable rate.

## Body

**Typ:** Specification  
**Layer:** Consensus (hard fork)  
**Status:** Closed (geschlossen (zurückgezogen oder ersetzt))  
**Themen:** bips
**Autor(en):** Gavin Andresen  
**Veröffentlicht:** 2015-06-22  
**Quelle:** https://github.com/bitcoin/bips/blob/master/bip-0101.mediawiki

### Zweck

BIP-0101 was Gavin Andresen's block size increase proposal — the most prominent of the block size debate era. It proposed starting at 8MB immediately (a hard fork), then doubling every two years until hitting a 8GB ceiling around 2036. The schedule tracked Andresen's projection of bandwidth and storage improvements.

This was implemented in Bitcoin XT and later Bitcoin Classic as a "hard fork" client, which saw significant miner signalling briefly in 2015 before support collapsed. The proposal failed not because the code didn't work, but because it couldn't achieve the community consensus needed for a hard fork. The controversy revealed that hard forks in Bitcoin require near-universal agreement — a much higher bar than anyone expected.

BIP-101's failure (and those of similar proposals) ultimately led to the SegWit path, which increased capacity through a soft fork.
