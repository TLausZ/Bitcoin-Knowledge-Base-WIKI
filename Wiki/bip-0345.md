# BIP-0345: OP_VAULT

## Summary

This proposal introduces OP_VAULT and OP_VAULT_RECOVER, two new opcodes enabling Bitcoin vaults — a custody mechanism where coins can be clawed back during a delay window if a compromise is detected.

**Layer:** Consensus (soft fork)

### Zweck

A Bitcoin vault addresses the worst-case custody scenario: an attacker who steals your key can immediately drain your funds. BIP-345 proposed two dedicated opcodes to enable a "trigger and delay" spending pattern.

`OP_VAULT` commits funds to a vault output that can only be spent in two ways: (1) trigger a withdrawal to a pre-specified target address, which enters a timelock delay, or (2) immediately sweep to a recovery address. During the delay period after a withdrawal is triggered, the recovery path remains available — allowing the owner to claw back funds if the withdrawal was unauthorised.

The vault setup works with a hot key (for daily access) and a cold recovery key. An attacker with the hot key can trigger a withdrawal, but the owner sees this on-chain and can invoke the recovery key before the delay expires. The recovery key needs only to be used in emergency, so it can be held in an extremely secure airgapped setup (or a multisig).

BIP-345 was closed in favour of BIP-443, which achieves the same vault semantics using a different opcode approach (OP_CTV + OP_VAULT in a combined proposal).
