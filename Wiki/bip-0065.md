# BIP-0065: OP_CHECKLOCKTIMEVERIFY

## Summary

This BIP describes a new opcode (OP_CHECKLOCKTIMEVERIFY) for the Bitcoin scripting system that allows a transaction output to be made unspendable until some point in the future.

### Zweck

`OP_CHECKLOCKTIMEVERIFY` (CLTV) allows a script to enforce that an output cannot be spent until a certain block height or Unix timestamp. Without CLTV, time locks could only be enforced at the transaction level (using `nLockTime`) — but those locks were set by the spender, not the creator of the output, so they offered no guarantee to the recipient.

CLTV puts the time lock in the output script itself. The opcode checks that the spending transaction's `nLockTime` is at least as large as the value on the stack. If not, the script fails. This enables constructions like: "this UTXO cannot be spent before block 900000" or "funds locked until January 2026."

Key use cases: timelocked inheritance setups, payment channel expiry conditions (e.g. Lightning Network), vaults, and fidelity bonds (BIP-46). CLTV is complemented by BIP-68's OP_CHECKSEQUENCEVERIFY, which enforces *relative* time locks ("cannot spend until N blocks after this output was confirmed").
