# BIP-0349: OP_INTERNALKEY

## Summary

This BIP proposes OP_INTERNALKEY, an opcode that pushes the Taproot internal key onto the stack during script execution.

## Body

**Typ:** Specification  
**Layer:** Consensus (soft fork)  
**Status:** Draft (noch im Entwurfsstadium)  
**Autor(en):** Brandon Black, Jeremy Rubin  
**Veröffentlicht:** 2024-11-14  
**Quelle:** https://github.com/bitcoin/bips/blob/master/bip-0349.md

### Zweck

In a Taproot script-path spend (BIP-341), the script being executed knows the Merkle root and the leaf content, but it doesn't have access to the internal key P that was used to compute the output key Q = P + t·G. OP_INTERNALKEY pushes P onto the stack.

This may seem like a minor convenience, but it allows a script to verifies a signature from "whoever controls this Taproot output's internal key" without hardcoding the key in the script leaf. The same script template can work with any internal key, making script reuse cleaner.

More importantly, when combined with CHECKSIGFROMSTACK (BIP-348), OP_INTERNALKEY enables covenant constructions where the internal key (which might be a MuSig2 aggregate of multiple parties) can sign off on spending conditions — delegating policy to the key holder without pre-committing the key in the script.
