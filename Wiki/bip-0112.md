# BIP-0112: CHECKSEQUENCEVERIFY

## Summary

This BIP describes a new opcode (CHECKSEQUENCEVERIFY) for the Bitcoin scripting system that in combination with BIP-68 allows execution pathways of a script to be restricted based on the age of the output being spent.

## Body

**Typ:** Specification  
**Layer:** Consensus (soft fork)  
**Status:** Deployed (aktiviert und im Einsatz)

### Zweck

`OP_CHECKSEQUENCEVERIFY` (CSV) is the script-layer counterpart to BIP-68's `nSequence` relative time locks. BIP-68 defined what `nSequence` values mean; BIP-112 makes those values enforceable within scripts.

When a script contains `<n> OP_CHECKSEQUENCEVERIFY`, it fails unless the spending transaction's input has an `nSequence` value indicating the UTXO has been "aged" for at least `n` blocks (or seconds, depending on the flag). This is a *relative* timelock — not "don't spend until block 900,000" but "don't spend until 144 blocks after this UTXO was confirmed."

CSV and BIP-68 activated together in July 2016. They are foundational to the Lightning Network: in a payment channel, commitment transactions use CSV delays to give a cheated party time to broadcast a penalty transaction before the cheater can claim funds. Without CSV, Lightning's security model doesn't work.

CSV is also used in atomic swaps, submarine swaps, and any protocol where a participant needs time to respond to a counterparty's on-chain action.
