# BIP-0320: nVersion bits for general purpose use

## Summary

This BIP reserves 16 bits of the block header nVersion field for general purpose use by miners, removing their meaning for version bits soft-fork signalling.

### Zweck

BIP-9 uses bits in the nVersion field for soft fork activation signalling. The field is 32 bits; BIP-9 uses bits 0–28 for signalling (bit 31 must be 0, bits 29–30 reserved). BIP-320 proposes reserving bits 16–30 explicitly for "overt AsicBoost" / version rolling — nonce space that miners can freely vary to extend their search space beyond the 32-bit nonce.

Without this reservation, miners rolling nVersion bits risk accidentally signalling for a soft fork they don't intend to support. BIP-320 makes the version rolling bits explicit and safe. BIP-323 (2026 Draft) extends this to 24 reserved bits.

The "overt AsicBoost" connection: AsicBoost is a mining optimisation that exploits SHA-256 internals; overt AsicBoost requires varying the nVersion field, which is why this intersection of mining efficiency and protocol signalling needed clarification.
