# BIP-0037: Connection Bloom filtering

## Summary

This BIP adds new support to the peer-to-peer protocol that allows peers to reduce the amount of transaction data they are sent. Peers have the option of setting filters on each connection they make after the version handshake has completed.

### Zweck

BIP-0037 introduced bloom filters for SPV (Simplified Payment Verification) light clients. An SPV client doesn't have a full copy of the blockchain — it only stores block headers and relies on full nodes for transaction data. Without BIP-37, an SPV client would have to download every transaction from a full node and filter locally, which is bandwidth-intensive.

With BIP-37, an SPV client sends a bloom filter (a probabilistic data structure) to a full node, encoding the set of addresses or scripts it cares about. The full node uses this filter to send only matching transactions. False positives in the filter are deliberate: they hide exactly which addresses the client is watching, providing some privacy against the full node.

However, BIP-37 was found to have serious problems. The bloom filter reveals significant information about the client's addresses — sophisticated observers can narrow down the exact addresses being tracked. More critically, it introduced DoS vectors: malicious peers could send crafted filters causing nodes to do excessive work. Bitcoin Core has since disabled BIP-37 by default. The recommended successor is BIP-157/158 (compact block filters), which shifts the filtering work back to the client.
