# BIP-0100: Dynamic maximum block size by miner vote

## Summary

Replace the static 1MB block size hard limit with a hard limit set by coinbase vote, conducted on the same schedule as difficulty retargeting.

## Body

**Typ:** Specification  
**Layer:** Consensus (hard fork)  
**Status:** Closed (geschlossen (zurückgezogen oder ersetzt))  
**Autor(en):** Jeff Garzik, Tom Harding, Dagur Valberg Johannsson  
**Veröffentlicht:** 2015-06-11  
**Quelle:** https://github.com/bitcoin/bips/blob/master/bip-0100.mediawiki

### Zweck

BIP-0100 was one of several block size increase proposals submitted during the 2015–2017 block size debate. It proposed replacing Bitcoin's 1MB block size limit with a miner-voted dynamic limit. Miners would vote for a desired block size in their coinbase transactions every 2016 blocks (one difficulty period); the median of votes would become the new limit, subject to a hard cap of 32MB.

The approach gave miners ongoing control over block size, subject to an upper bound. Critics argued this granted miners too much governance power over a parameter that affects all users' costs to run full nodes. The proposal was closed without adoption when SegWit emerged as the preferred path — SegWit increased effective transaction throughput without a hard fork and without delegating block size governance to miners.

## Open Questions

- Should miners have any vote over block size, or should it be strictly a user/developer/node-operator decision?
