# BIP-0017: OP_CHECKHASHVERIFY (CHV)

## Summary

This BIP describes a new opcode (OP_CHECKHASHVERIFY) for the Bitcoin scripting system, enabling the receiver of bitcoins to specify the transaction type needed to re-spend them.

## Body

**Typ:** Specification  
**Layer:** Consensus (soft fork)  
**Status:** Closed (geschlossen (zurückgezogen oder ersetzt))  
**Autor(en):** Luke Dashjr  
**Veröffentlicht:** 2012-01-18  
**Quelle:** https://github.com/bitcoin/bips/blob/master/bip-0017.mediawiki

### Zweck

BIP-0017 was a competing proposal to BIP-16 (P2SH) during the 2012 script-hashing debate. Where BIP-16 achieved script hashing by repurposing a special output pattern (`OP_HASH160 <hash> OP_EQUAL`), BIP-17 proposed a new opcode `OP_CHECKHASHVERIFY` that would explicitly verify a hash commitment in the scriptPubKey.

Luke Dashjr's argument for BIP-17 was that BIP-16 was a "hack" — it repurposed an existing pattern in a semantically confusing way. BIP-17 would have been a cleaner opcode-based extension. The counterargument was that adding new opcodes is a harder consensus change than repurposing existing script behaviour.

BIP-16 activated in April 2012, making BIP-17 moot. The philosophical tension between "clean design" and "minimal footprint" that this dispute represents has recurred in later Bitcoin design debates.
