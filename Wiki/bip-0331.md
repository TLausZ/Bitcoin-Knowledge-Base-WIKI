# BIP-0331: Ancestor Package Relay

## Summary

Peer-to-peer protocol messages enabling nodes to request and relay the unconfirmed ancestor package of a given transaction, and to request and relay transactions in batches.

### Zweck

Bitcoin's mempool evaluates transactions individually for minimum fee rate. If a high-fee child transaction depends on a low-fee parent that hasn't been relayed yet, nodes will reject the parent (too low fee) and never see the child. Child-pays-for-parent (CPFP) fee bumping only works if the parent is already in the mempool.

Package relay (BIP-331) solves this by allowing nodes to relay related transactions together as a package and evaluate them as a unit. A transaction paying 0 sat/vbyte could be relayed as part of a package whose combined fee rate meets the minimum, because the child's fees subsidise the parent.

Lightning Network channel opens depend on this: their commitment transaction fees are set at channel creation and can't be adjusted. If fees rise significantly, the pre-signed commitment transaction may be too low-fee to relay and confirm, preventing channel closure. Package relay lets a CPFP child rescue a stuck parent.

New P2P messages: `sendpackages`, `getpckgtxns`, `pckgtxns`, `ancpkginfo`. Nodes negotiate package relay support during handshake and can request ancestor packages for transactions they receive.
